# 卡密唯一性保证机制

## 核心原则

**一张卡密只能被使用一次，无论谁使用后立即失效，并立即写入数据库，绝不允许再次兑换。**

## 关键修复（必读）

**应用层禁止清空 `recharge_cards.used_by`。**

历史上若在「解析失败 / 余额写入失败」时调用「回滚」把 `used_by` 置空，**同一张卡会被系统再次视为未使用**，用户可反复充值——这是严重漏洞。

正确行为：

- `atomic_redeem_card` **一旦成功**，该卡即 **永久核销**；
- 若后续解析次数或写余额失败，**不得**自动把 `used_by` 清空；
- 只能由管理员在数据库中人工处理，并提示用户 **勿重复输入同一卡密**。

## 实现机制

### 1. 数据库：`atomic_redeem_card`

见 `sql/atomic_redeem_card.sql`：单条 `UPDATE ... WHERE used_by 为空`，成功则写入 `used_by` / `used_at`，失败则返回 NULL。

### 2. 应用：仅通过 RPC 核销

`src/routes/recharge.ts` 中所有兑换均走 `atomicRedeemCard` → `supabase.rpc('atomic_redeem_card', ...)`，**无**其它路径直接改 `used_by` 为「已使用」后再清空。

### 3. 不再有「释放卡密」逻辑

已删除对 `recharge_cards` 的「回滚」更新（将 `used_by` 置 NULL）。

## 测试建议

1. 同一卡密连续兑换两次：第二次必须提示 **卡密已失效**（或等价文案），且余额只应增加一次。
2. 模拟余额写入失败：卡密仍应保持已使用，**不能**再次兑换成功。
